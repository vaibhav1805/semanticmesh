<objective>
Extract query execution logic from CLI handlers into exported, reusable functions that both CLI and MCP layers can call.

Purpose: The MCP server (Plan 02) needs to call query logic from a separate package (`internal/mcp/`). Currently, the core traversal logic is either unexported (`executeImpactReverse`, `executeForwardTraversal`) or embedded in CLI handler functions that parse flags and write to stdout. This plan creates a clean API boundary.

Output: `query_exec.go` with 5 exported functions (impact, dependencies, path, list, graph_info) accepting typed param structs and returning `(*QueryEnvelope, error)`. CLI handlers refactored to delegate to these functions.
</objective>

  <action>
Create `internal/knowledge/query_exec.go` with these typed param structs and exported functions:

1. **Param structs** (all fields match CLI flags):
   - `QueryImpactParams{Component, Depth int, MinConfidence float64, SourceType, GraphName string, IncludeProvenance bool, MaxMentions int}`
   - `QueryDependenciesParams` — identical fields to QueryImpactParams
   - `QueryPathParams{From, To string, Limit int, MinConfidence float64, SourceType, GraphName string}`
   - `QueryListParams{TypeName string, MinConfidence float64, SourceType, GraphName string}`
   - `GraphInfoParams{GraphName string}`

2. **GraphInfoResult struct** for graph_info responses:
   - `Name string, Version string, CreatedAt string, ComponentCount int, RelationshipCount int, SchemaVersion int`

3. **ExecuteImpactQuery(params QueryImpactParams) (*QueryEnvelope, error)**:
   - Validate `params.Component != ""` (return error with MISSING_ARG-style message)
   - Validate source type via `validateSourceType`
   - Default depth to 1 if <= 0
   - Call `LoadStoredGraph(params.GraphName)`
   - Check component exists; if not return structured error (use new `queryError` type that wraps code+message+suggestions, distinct from writing to stdout)
   - Call `executeImpactReverse` with the params
   - Optionally decorate with provenance if `IncludeProvenance`
   - Build and return `QueryEnvelope` — do NOT write anything to stdout

4. **ExecuteDependenciesQuery(params QueryDependenciesParams) (*QueryEnvelope, error)**:
   - Same pattern as impact but calls `executeForwardTraversal`

5. **ExecutePathQuery(params QueryPathParams) (*QueryEnvelope, error)**:
   - Validate `From` and `To` not empty
   - Default limit to 10 if <= 0
   - Call `LoadStoredGraph`, validate components, call `g.FindPaths`, build hops, sort, apply limit
   - Return envelope (empty paths is not an error — include reason in PathResult)

6. **ExecuteListQuery(params QueryListParams) (*QueryEnvelope, error)**:
   - Same filtering logic as `cmdQueryList` but returns envelope
   - Type filter, min-confidence filter, source-type filter

7. **GetGraphInfo(params GraphInfoParams) (*GraphInfoResult, error)**:
   - Call `LoadStoredGraph(params.GraphName)`
   - Return `GraphInfoResult` with name, version, created_at, component count, edge count

8. **QueryError type** for structured error returns:
   - `type QueryError struct { Message string; Code string; Suggestions []string }`
   - Implement `error` interface: `func (e *QueryError) Error() string { return e.Message }`
   - MCP layer can type-assert to `*QueryError` to distinguish user errors (NOT_FOUND, MISSING_ARG) from infrastructure errors

All functions must be pure data-in/data-out: no `os.Stdout` writes, no `os.Exit`, no `flag` parsing. The `minConf` pointer pattern from the unexported functions should be converted to a float64 value (0 means no filter).

Create `internal/knowledge/query_exec_test.go` with tests:
- `TestExecuteImpactQuery_MissingComponent`: returns QueryError with code MISSING_ARG
- `TestExecuteImpactQuery_InvalidSourceType`: returns QueryError with code INVALID_ARG
- `TestExecuteDependenciesQuery_MissingComponent`: same pattern
- `TestExecutePathQuery_MissingArgs`: returns error when From or To empty
- `TestExecuteListQuery_InvalidSourceType`: returns error for bad source type
- `TestGetGraphInfo_NoGraph`: returns error when no graph loaded
- `TestQueryError_Interface`: QueryError satisfies error interface and is type-assertable

Note: Tests that require a loaded graph will get "no graph imported" errors, which is fine — they verify the error handling path. Tests for actual query execution against real data are covered by existing `query_cli_test.go`.

Refactor the 4 CLI handler functions in `query_cli.go` to delegate to the new exported functions:

1. **cmdQueryImpact**: Parse flags as before, then call `ExecuteImpactQuery(QueryImpactParams{...})`. On error, check if `*QueryError` and write error JSON to stdout (preserving existing behavior). On success, call `outputEnvelope`.

2. **cmdQueryDependencies**: Same pattern — parse flags, call `ExecuteDependenciesQuery`, handle errors, output.

3. **cmdQueryPath**: Parse flags, call `ExecutePathQuery`. Note: path queries always succeed (empty paths not an error), so use `outputEnvelopeSuccess`.

4. **cmdQueryList**: Parse flags, call `ExecuteListQuery`. Use `outputEnvelopeSuccess`.

The key behavior to preserve:
- Error JSON written to stdout for user errors (MISSING_ARG, NOT_FOUND, INVALID_ARG)
- Table format still works (outputEnvelope handles format routing)
- Exit codes unchanged (error return from handler triggers exit 1 in main.go)

Delete the inline query logic that is now in `query_exec.go` (graph loading, component validation, traversal calls, envelope construction). Keep only: flag parsing, error output formatting, and the call to Execute*.

Do NOT change: `outputEnvelope`, `outputEnvelopeSuccess`, `writeTable`, `printQueryUsage`, `writeErrorJSON`, `writeErrorJSONStdout`, `handleLoadError` — these remain as CLI-layer output utilities.
